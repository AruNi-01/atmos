### 🟠 P1 — High

- **`apps/web/src/app/api/review-skills/route.ts:6`** — Missing error handling for filesystem operations
  - **Category**: Error Handling
  - **Issue**: The API route uses `fs.existsSync()` and `fs.readdirSync()` without proper error handling. If the `.atmos` directory has permission issues or filesystem errors occur, the route will throw an unhandled exception and crash the request.
  - **Suggestion**: Wrap filesystem operations in try-catch and return appropriate error responses.
    ```typescript
    export async function GET() {
      try {
        const systemReviewSkillsPath = path.join(os.homedir(), '.atmos', 'skills', '.system', 'code_review_skills');
        const skills: any[] = [];
        // ... rest of code
      } catch (error) {
        console.error('Failed to load review skills:', error);
        return NextResponse.json({ skills: [] }, { status: 200 }); // Graceful fallback
      }
    }
    ```

- **`apps/web/src/components/code-review/CodeReviewDialog.tsx:176-186`** — Unhandled promise rejection in skill loading
  - **Category**: Error Handling
  - **Issue**: The fetch call to `/api/review-skills` catches errors but only logs to console. If the API endpoint fails (e.g., 404, 500), the UI will show an empty skills list without user feedback. The error is silently swallowed.
  - **Suggestion**: Add error state handling and user feedback.
    ```typescript
    const [skillsLoadError, setSkillsLoadError] = useState<string | null>(null);
    // ...
    .catch((err) => {
      console.error("Failed to load generic review skills", err);
      setSkillsLoadError("Failed to load review skills. Please refresh.");
    })
    // In JSX: {skillsLoadError && <div className="text-destructive text-xs">{skillsLoadError}</div>}
    ```

### 🟡 P2 — Medium

- **`apps/api/src/utils/system_skill_sync.rs:56`** — Race condition in temp directory creation
  - **Category**: Race Condition
  - **Issue**: The temp directory name uses `std::process::id()` which is generally unique, but between directory creation and git clone, another process could create the same directory. The `create_dir_all` error is silently ignored with `is_err()` check returning `false`.
  - **Suggestion**: Use `std::fs::create_dir` with error handling to detect actual conflicts, or use a UUID-based temporary directory.
    ```rust
    use uuid::Uuid;
    let temp_dir = std::env::temp_dir().join(format!("atmos-skill-sync-{}", Uuid::new_v4()));
    ```

- **`apps/web/src/components/agent/AgentChatPanel.tsx:1313-1325`** — Potential race condition with pending prompt handling
  - **Category**: Logic | Race Condition
  - **Issue**: Between peeking `pendingAgentChatPrompt()` and calling `startSession()`, another effect could consume the same prompt. The `consumePendingAgentChatPrompt()` at line 1391 happens after session connection, creating a window where the prompt could be lost or duplicated.
  - **Suggestion**: Consume the prompt immediately in this effect if it exists, or use a ref to track "processing" state.

- **`crates/core-service/src/service/ws_message.rs:1528-1559`** — Duplicated directory copy logic
  - **Category**: Maintainability | Duplicated Code
  - **Issue**: The `copy_dir_all` function is duplicated between `ws_message.rs` and `system_skill_sync.rs`. This violates DRY and creates maintenance burden.
  - **Suggestion**: Extract this to a shared utility module in `infra` or `core-engine`.

- **`apps/web/src/components/workspace/OverviewTab.tsx:165-177`** — Missing dependency in useEffect
  - **Category**: React | Hooks
  - **Issue**: The useEffect has `loadReviews` in the dependency array but `loadReviews` is recreated on every render (not wrapped in `useCallback`). This could cause infinite re-renders or unnecessary API calls.
  - **Suggestion**: Wrap `loadReviews` in `useCallback` with proper dependencies.
    ```typescript
    const loadReviews = useCallback(async () => {
      // ... existing code
    }, [projectPath, contextId]); // Add dependencies
    ```

- **`apps/web/src/hooks/use-dialog-store.ts:55-70`** — Non-standard Zustand pattern for peek/consume
  - **Category**: Code Smell | State Management
  - **Issue**: The `peekPendingAgentChatPrompt` function creates unnecessary state updates by calling `set()` with no-op state, causing all subscribers to re-render. Zustand's `getState()` should be used instead.
  - **Suggestion**: Use `useDialogStore.getState()` for peek operations without triggering updates.
    ```typescript
    peekPendingAgentChatPrompt: () => {
      return get().pendingAgentChatPrompt;
    },
    ```

- **`apps/web/src/components/diff/DiffViewer.tsx:226-247`** — Brittle DOM querying for shadow root elements
  - **Category**: Maintainability | DOM Manipulation
  - **Issue**: The code queries for `diffs-container` and shadow root elements using string selectors that depend on `@pierre/diffs` internal structure. If the library updates its DOM structure, this will break.
  - **Suggestion**: Consider using the library's built-in positioning APIs if available, or add a comment noting this is a fragile integration point.

- **`apps/web/src/components/code-review/CodeReviewDialog.tsx:392-394`** — In-line styles in JSX
  - **Category**: Style | Code Smell
  - **Issue**: `style={{ gridTemplateColumns: '1fr 1fr' }}` is an in-line style. For consistency with the codebase's Tailwind usage, consider using a utility class or a styled component.
  - **Suggestion**: Use Tailwind's arbitrary grid syntax: `className="grid grid-cols-2"`.

- **`packages/ui/src/styles/globals.css:127-147`** — Animation naming collision risk
  - **Category**: CSS | Maintainability
  - **Issue**: The `@keyframes collapsible-down` and `collapsible-up` animations are defined globally. If other components define similar animations, there could be conflicts.
  - **Suggestion**: Consider using CSS modules or Tailwind's built-in animations (which the codebase already uses via `tw-animate-css`).

## Architecture Notes

### Code Organization

The changes demonstrate good separation of concerns:
- **Backend**: Skill sync logic properly isolated in `system_skill_sync.rs`
- **Frontend**: Code review dialog follows existing patterns with `AgentSelect`
- **API**: Clean REST endpoint for skill discovery

### Dependencies

The code properly follows the monorepo conventions:
- `@workspace/ui` imports are consistent
- API client patterns (`ws-api`, `rest-api`) are followed
- No circular dependencies detected

### Potential Coupling Issues

1. **Tight coupling to `@pierre/diffs`**: The `DiffViewer` component heavily relies on internal DOM structure of the diff library. Consider adding an abstraction layer or contributing positioning APIs upstream.

2. **Skill path assumptions**: Multiple places assume the `~/.atmos/skills/.system/code_review_skills/` path structure. A centralized constants module would reduce coupling.

# ADR-001: 采用 Monorepo 架构

**状态**: ✅ 已采纳  
**日期**: 2024-01-10  
**决策者**: @alice (架构师), @bob (技术负责人)  
**相关人员**: @charlie (前端), @david (后端), @eve (DevOps)

## 背景 (Context)

我们的项目正在快速发展,目前包含以下组件:

- **Web 应用**: Next.js 前端,面向最终用户
- **文档站点**: 产品文档和 API 参考
- **桌面应用**: Tauri 跨平台桌面客户端(计划中)
- **CLI 工具**: 命令行工具,供开发者和 AI 使用
- **共享组件**: UI 组件库、工具函数等

**当前问题**:

1. **代码分散**: 各个项目在独立仓库,导致:
   - 共享代码需要发布到 npm,流程繁琐
   - 依赖版本不一致(Web 用 React 18,Docs 用 React 17)
   - 跨项目重构困难(需要协调多个 PR)

2. **协作效率低**:
   - 开发者需要在多个仓库间切换
   - CI/CD 配置重复
   - Issue 和 PR 分散在不同仓库

3. **发布复杂**:
   - 需要协调多个仓库的版本
   - 难以保证各组件的兼容性

**约束条件**:

- 团队规模: 3-5 人
- 主要技术栈: TypeScript (前端) + Rust (后端/CLI)
- 希望保持快速迭代能力

## 决策 (Decision)

我们决定**采用 Monorepo 架构**,将所有项目整合到一个 Git 仓库中。

**技术方案**:
- **包管理器**: Bun workspaces (快速、现代)
- **构建工具**: 原生工具 + Just (跨语言任务编排)
- **版本管理**: 手动管理(暂不使用 Changesets,避免过度工程)

**目录结构**:
```
my-project/
├── apps/           # 可部署的应用
│   ├── web/
│   ├── docs/
│   ├── desktop/
│   ├── cli/
│   └── api/
├── packages/       # 前端共享包
│   ├── ui/
│   ├── shared/
│   └── config/
├── crates/         # Rust 共享库
│   ├── core/
│   └── common/
└── docs/           # 开发文档
```

## 考虑的方案 (Alternatives Considered)

### 方案 1: 多仓库 (Polyrepo) - 现状

**描述**: 每个项目独立一个 Git 仓库

**优点**:
- 仓库职责单一,边界清晰
- 权限控制简单(可以按仓库分配权限)
- Git 历史简洁,单个仓库体积小
- 可以独立设置 CI/CD

**缺点**:
- 代码共享困难,需要发布 npm 包
- 依赖版本管理复杂,容易不一致
- 跨项目重构需要协调多个 PR
- CI/CD 配置重复,维护成本高
- 新人上手需要克隆多个仓库

**评估**: ❌ 不适合我们的场景
- 我们的组件高度耦合,需要频繁跨项目修改
- 团队规模小,权限控制不是主要问题
- 发布 npm 包的成本太高

---

### 方案 2: Monorepo

**描述**: 所有项目在一个仓库,使用 workspace 管理

**优点**:
- **代码共享简单**: 直接 `import` 内部包,无需发布
- **原子性提交**: 跨项目修改一次 commit 搞定
- **依赖统一管理**: 所有项目使用相同版本的工具
- **重构友好**: IDE 可以跨项目重构
- **单一入口**: 新人只需克隆一个仓库
- **CI/CD 统一**: 一套配置管理所有项目

**缺点**:
- 仓库体积变大(预计 500MB - 1GB)
- Git 操作可能变慢(可用 sparse-checkout 缓解)
- 需要良好的目录结构规范
- 需要学习 workspace 工具

**评估**: ✅ **选择此方案**
- 优点完全契合我们的需求
- 缺点可以通过工具和规范缓解

---

### 方案 3: Git Submodules

**描述**: 主仓库包含多个子仓库作为 submodule

**优点**:
- 保持仓库独立性
- 可以选择性克隆
- 某种程度上实现代码组合

**缺点**:
- 使用复杂,团队成员容易出错
- 依赖管理仍然困难
- 子模块更新流程繁琐
- IDE 支持不佳

**评估**: ❌ 不推荐
- Submodule 有很多坑,学习成本高
- 并没有解决依赖管理问题

---

### 方案 4: Meta Repository (Lerna/Rush)

**描述**: 使用 Lerna 等工具管理多个独立仓库

**优点**:
- 每个包仍是独立仓库
- 提供统一的发布工具
- 适合发布到 npm 的场景

**缺点**:
- 工具复杂,配置繁琐
- 本质上还是多仓库,共享代码困难
- Lerna 已不再积极维护

**评估**: ❌ 不适合
- 我们大部分包是内部使用,不需要发布

## 后果 (Consequences)

### 正面影响 ✅

1. **开发效率提升**:
   - 跨项目重构时间减少 60%
   - 新增共享组件只需几分钟(vs 之前需要发布 npm 包)
   - IDE 智能提示可以跨项目工作

2. **代码质量提升**:
   - 强制统一代码规范(共享 ESLint/Prettier 配置)
   - 依赖版本一致,减少兼容性问题
   - 更容易做全局性的改进

3. **协作体验改善**:
   - 所有 Issue/PR 在一个地方
   - Code Review 可以看到完整上下文
   - 新人上手时间从 2 天减少到 0.5 天

4. **部署简化**:
   - 一次性构建所有项目
   - 确保各组件版本兼容
   - 减少发布协调成本

### 负面影响 ⚠️

1. **仓库体积增大**:
   - 预计从 50MB × 5 → 500MB 单仓库
   - Git clone 时间增加(约 30 秒 → 2 分钟)

2. **学习成本**:
   - 团队需要学习 workspace 概念
   - 需要理解 Monorepo 目录结构

3. **Git 操作稍慢**:
   - `git status` 等命令在大仓库可能变慢
   - 历史记录变长

4. **权限控制粒度降低**:
   - 无法按单个项目分配权限
   - 所有人都能访问所有代码

### 风险和缓解措施 🛡️

| 风险 | 严重程度 | 概率 | 缓解措施 |
|------|---------|------|---------|
| 仓库过大导致 Git 性能问题 | 中 | 中 | 使用 Git LFS 存储大文件;启用 sparse-checkout;定期清理历史 |
| 新人不理解 Monorepo 结构 | 低 | 高 | 编写详细的开发文档;提供示例和最佳实践 |
| CI/CD 构建时间过长 | 中 | 低 | 使用 Turbo 增量构建;CI 缓存依赖;并行构建 |
| 某个项目的错误影响整个仓库 | 低 | 低 | 严格的 CI 检查;代码审查;使用 husky 预提交钩子 |

### 中立影响 ℹ️

- 需要建立 Monorepo 的最佳实践和规范
- CI/CD 配置需要一次性调整
- 所有开发者需要学习新的工作流程
